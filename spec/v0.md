# Wire-Format Protocol Specification — v0 (Draft)

Status: **draft**, pending solution-architect and security-engineer sign-off (§8). This is the
Phase 0 "protocol spec v0 draft" deliverable from `PLAN.md` §7 / §5 ("Protocol portability").
Schemas referenced here live in `spec/proto/v0/*.proto` and are the source of truth for field
numbers and types; this document is the rationale and the rules that govern how those schemas
are used and evolved.

This spec does not re-derive or restate the Signal Protocol itself. X3DH/PQXDH, the Double
Ratchet, and Sender Keys are taken as given from upstream `libsignal`, exactly as
`docs/threat-model.md` does. What this spec defines is the layer **around** `libsignal`: how its
opaque ciphertext is framed, addressed, and routed between a sender, the transport (direct P2P or
a relay), and a recipient — the "Protocol layer" in `docs/threat-model.md` §4.2.

---

## 1. Scope

In scope for v0:
- The outer `Envelope` that every message, receipt, and Sender Key distribution travels in.
- The Sealed Sender wrapper structure.
- Prekey bundle and DHT discovery record shapes used for asynchronous session setup.
- The versioning and compatibility rules that govern all of the above.

Out of scope for v0 (tracked separately, not silently assumed):
- The internal wire format of `libsignal`'s own messages (`PreKeySignalMessage`,
  `SignalMessage`, `SenderKeyMessage`, `SenderKeyDistributionMessage`). Those are upstream
  `libsignal` types we treat as opaque bytes — see §4.
- Relay-to-relay or relay-to-DHT control protocols (fetch/store APIs, rate-limiting handshakes).
  This spec defines what a relay stores and forwards, not how a client talks to a specific
  relay's API — that is Phase 4 work (`PLAN.md` §7).
- Backup/export file format (`PLAN.md` §5, data portability) — a separate spec.
- Device-linking message shapes (`PLAN.md` §4) — drafted alongside the Phase 6 Multi-Device epic.
- Actual `prost`/codegen wiring into `core/protocol` — this story is the spec draft only; wiring
  generated types into the Rust crate is separate implementation work so review of the wire
  format itself isn't coupled to a specific codegen toolchain choice.

## 2. Design Principles

1. **The envelope never inspects Signal Protocol ciphertext.** `Envelope.ciphertext` (and the
   `inner_ciphertext` it may wrap via Sealed Sender) is opaque bytes produced and consumed only
   by `libsignal`. This protocol layer is not in the business of cryptography — it is in the
   business of addressing and framing, per `PLAN.md` §2 layer responsibilities.
2. **Routing metadata is the metadata budget, and it is itemized.** Every plaintext field on
   `Envelope` is something a relay or DHT peer can see (`docs/threat-model.md` §4.2,
   §6 "Summary: What Leaks Where"). A field is only added if something downstream actually needs
   it for routing or delivery; "might be useful later" is not sufficient justification for a
   field that a malicious relay can also read.
3. **Fail closed on the unknown.** Protobuf's forward-compatibility (unknown fields are
   preserved/ignored, not rejected) is necessary but not sufficient: an `EnvelopeType` a client
   doesn't recognize, or a `version` newer than it supports, MUST be treated as undeliverable to
   the application layer, never as a best-effort fallback to some default decoding. See §3.
4. **Sender identity and routing hints are never both plaintext at once for content messages.**
   `Envelope.sender` is unset whenever `ciphertext` is a Sealed-Sender-wrapped content message —
   see §5. This is enforced by convention in this draft; a follow-up story should turn it into a
   structural invariant (e.g. a oneof) once the surrounding session-management code exists to
   validate it meaningfully.

## 3. Versioning Policy

Two independent version numbers exist and must not be conflated:

- **`Envelope.version`** (`spec/proto/v0/envelope.proto`) — the wire version of this outer
  envelope format. This document defines version `0`.
- **The Signal Protocol session/ciphertext version** carried inside `libsignal`'s own serialized
  messages — entirely libsignal's concern, opaque to this layer (§1, §4).

Rules for v0:
- A v0 client sets `Envelope.version = 0` and MUST reject (fail closed, never best-effort parse)
  any envelope whose `version` it does not explicitly know how to handle.
- New fields within version `0` follow standard protobuf evolution: new fields get new field
  numbers, are optional in effect (proto3 scalar fields default to their zero value), and old
  clients ignore fields they don't recognize. This is "non-breaking" only in the wire-decoding
  sense — a v0 client still cannot act on semantics it doesn't understand.
- A **breaking** change (removing a field, changing a field's meaning, changing a field's type,
  or changing required behavior for an existing `EnvelopeType`) requires incrementing
  `Envelope.version` and a new dated entry in §9 (Changelog), not a silent edit of this document.
- This document and the `.proto` files it describes are versioned together. `spec/v0.md` and
  `spec/proto/v0/` will not change incompatibly once any client implements against them; the next
  incompatible revision is `spec/v1.md` / `spec/proto/v1/`, published alongside, not in place of,
  v0.
- "Published" (per `PLAN.md` §5, "Protocol portability") means: once this draft is signed off
  (§8) and a client ships against it, `spec/proto/v0/*.proto` is frozen. Independent
  implementations can rely on it not moving under them.

## 4. Mapping `EnvelopeType` to `libsignal` Message Types

| `EnvelopeType` | `ciphertext` / `inner_ciphertext` contains | Used for |
|---|---|---|
| `ENVELOPE_TYPE_PREKEY_MESSAGE` | `libsignal` `PreKeySignalMessage` | First message of a new X3DH/PQXDH session (`PLAN.md` §3) |
| `ENVELOPE_TYPE_CIPHERTEXT_MESSAGE` | `libsignal` `SignalMessage` | Double Ratchet message in an established 1:1 session |
| `ENVELOPE_TYPE_SENDER_KEY_MESSAGE` | `libsignal` `SenderKeyMessage` | Group message under an established Sender Key session (`PLAN.md` "Locked decisions") |
| `ENVELOPE_TYPE_SENDER_KEY_DISTRIBUTION` | `libsignal` `SenderKeyDistributionMessage` | Distributing/rotating a Sender Key to a group member |
| `ENVELOPE_TYPE_RECEIPT` | `libsignal` `SignalMessage` (a receipt is itself a Double-Ratchet-encrypted 1:1 message; only its decrypted application-layer content differs) | Delivery/read acknowledgement |

This layer never decodes the bytes in these columns; it routes them. The mapping exists so an
implementer knows which `libsignal` deserializer to hand the bytes to once `Envelope.type` and
Sealed Sender status (§5) identify which case applies.

## 5. Sealed Sender

`spec/proto/v0/sealed_sender.proto` defines `SealedSenderContent` — the structure of the data
that gets Sealed-Sender-encrypted by `libsignal`, not a structure that appears in plaintext on
the wire. When Sealed Sender is in effect for an envelope:

- `Envelope.sender` is unset.
- `Envelope.ciphertext` holds libsignal's sealed-sender-encrypted bytes, which decrypt (only by
  the intended recipient) to a `SealedSenderContent`.
- `SealedSenderContent.sender` is the real sender device address, `sender_certificate` is a
  short-lived self-signed certificate binding that address to the sender's identity key (no
  central CA exists in this design — `PLAN.md` §3), and `inner_ciphertext`/`inner_type` carry the
  actual `libsignal` message per §4.

Per `docs/threat-model.md` §4.1: this hides the sender field from anyone who handles the
envelope in transit, but it does **not** hide the sender's network identity (source IP / libp2p
peer ID) from a relay, and it provides no protection against a compromised recipient. Treat it as
a confidentiality control on one field, not a network-anonymity guarantee.

Sealed Sender is **not** used for `ENVELOPE_TYPE_SENDER_KEY_DISTRIBUTION` or
`ENVELOPE_TYPE_RECEIPT` in v0: group membership already discloses participants for the former,
and receipts travel inside an already-established 1:1 session where the recipient already knows
the sender by definition. Extending Sealed Sender to receipts is an open item (§7), not a gap
being silently accepted — receipts currently disclose `Envelope.sender` in plaintext to the relay
they transit, which is a real (if narrow) metadata leak relative to a normal message.

## 6. Prekeys and Discovery

`spec/proto/v0/prekey.proto` defines:

- `SignedPreKey` / `OneTimePreKey` / `PqPreKey` — the medium-term, one-time, and optional
  post-quantum components of a session-setup bundle, matching `PLAN.md` §3.
- `PreKeyBundle` — what a sender fetches to start a session with an offline device. Every
  signature in a fetched bundle **must** be verified against `identity_key` before use; per
  `docs/threat-model.md` §4.6, a DHT response is a hint, not a fact, until verified.
- `DiscoveryRecord` — what gets published to the DHT under a handle: one `PreKeyBundle` per
  linked device, signed as a set by the primary device so a malicious DHT peer can't splice in a
  rogue device's bundle (`docs/threat-model.md` §2, "Handle-squatter" and "Malicious DHT peer"
  adversaries). This signature does not, and cannot, defend against a primary device that is
  itself compromised — that adversary already holds the signing key by definition.

An empty `one_time_pre_keys` list is a valid, expected state (pool exhaustion under load), not an
error condition; see the field comment in `prekey.proto` and the corresponding risk entry in
`PLAN.md` §10 ("Offline prekey exhaustion").

## 7. Open Items Deferred Past v0

Tracked here so they are visible, not silently dropped:

- **Receipt metadata leak** (§5): `ENVELOPE_TYPE_RECEIPT` currently discloses `Envelope.sender`
  in plaintext to the relay. Revisit once Sealed Sender's certificate model is validated for the
  1:1 message case it already covers.
- **No padding/length-hiding.** `Envelope.ciphertext` length leaks approximate message size to
  every relay/DHT peer that handles it, consistent with the residual "traffic-analysis metadata"
  risk in `docs/threat-model.md` §6. Padding is explicitly a Phase 9 item (`PLAN.md` §7).
  Reserving a field for it now (e.g. a fixed-size padding blob) is deferred rather than guessed
  at speculatively in v0 — see Core Principles ("minimal footprint") in `CLAUDE.md`.
  - **Field-number reservation:** `Envelope` reserves no field for padding in v0. If a v1
    revision adds one, it must use a new field number (per §3) so v0 decoders that ignore
    unknown fields remain correct rather than misinterpreting padding bytes as something else.
- **Relay control-plane protocol** (§1): how a client authenticates to, queries, and pushes
  envelopes to a specific relay instance is not specified here. Phase 4 (`PLAN.md` §7) work.
- **Multi-device fan-out addressing for groups**: `ENVELOPE_TYPE_SENDER_KEY_MESSAGE` addresses a
  single `DeviceAddress` recipient per envelope (sender fan-out per recipient device, per
  `PLAN.md` §4); this spec does not define a batched/multi-recipient envelope shape. Whether v1
  needs one is an open question for the Phase 7 Groups epic, not decided here.
- **`prost`/codegen wiring into `core/protocol`**: out of scope per §1; tracked as follow-up
  implementation work once this draft is signed off.
